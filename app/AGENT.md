Golang Clean Architecture & Quality Standards
1. Core PrinciplesClean Architecture: Enforce strict separation between Entities, Usecases, and Interfaces.Dependency Rule: Dependencies must only point inwards. High-level logic (Usecases) must not depend on low-level details (Databases/APIs).SOLID: Prioritize Single Responsibility (SRP) and Dependency Inversion.Serverless Mindset: All logic must be stateless and event-driven where possible.

2. Directory Structure (Clean Archi)Plaintext/internal

3. Implementation RulesA. Repository & Data LayerInterface Driven: Define a Repository interface in the Usecase layer; implement it in the Adapter layer.NoSQL Chunking: For the canvas, implement chunked partitioning (e.g., 100x100) to ensure "infinite" scalability.Atomic Updates: Use database-native atomic counters for rate limiting (e.g., ADD in DynamoDB) to prevent race conditions.B. Usecase & Business LogicSingle Responsibility: One function/service = one clear task.Asynchronous Processing: Long-running tasks (Snapshots/Broadcasting) must be handled via message brokers (SNS/SQS).Validation: Validate all inputs at the entry point before passing them to the Usecase.C. Security & ObservabilityLeast Privilege: Assign specific IAM roles/policies per service.Secrets: Never hardcode credentials; use AWS Secrets Manager or equivalent.Structured Logging: All logs must be in JSON format and include a correlation_id and trace_id.Tracing: Implement distributed tracing to monitor request flow across services.4. Error Handling StandardDon't Panic: Never use panic() in production code.Wrap Errors: Use %w to wrap errors for context while maintaining traceability.Fail-Closed: If authentication or validation fails, default to a "Deny" or "Error" state.5. Definition of Done (DoD)Business Logic is 100% independent of the UI and Database.IAM Permissions are scoped to the minimum required actions.Unit Tests cover the Usecase layer with mocked adapters.Logging is instrumented for every key event (Draw, Auth, Error).
